# 「制御して使う」開発パートナーとしての Copilot

## 気づき

GitHub Copilot はもはや単なるコード補完ツールではない。

> 「書かせている」状態から、「**制御して使っている**」状態へ

どの文脈を与え、どこまで任せ、どこで人が介入するか。  
その設計次第で、生産性にもコード品質にも大きな差が生まれる。

## 「ええ感じにやっといて」が実現する条件

上司から言われる「ええ感じにやっといて」を Copilot に実現させるには、以下の3点が必要：

1. **コンテキストの指定** — #codebase / #file / #git 等で参照情報を与える
2. **ツールの準備** — 組み込み機能・MCP で実行できる機能を整える
3. **カスタム命令** — AGENTS.md / instructions.md でルールを定義する

この3点が揃って初めて「指示が多少曖昧でも方針がブレずにタスクを遂行してくれる」状態になる。

## AI 駆動開発での役割分担

| 人間がやること | Copilot がやること |
|---|---|
| Issue・仕様の作成 | Issue を読んで自律的に実装 |
| カスタム命令・コンテキストの整備 | ルールに従ってコード生成 |
| PR のマージ判断 | PR の自動作成・コードレビュー |
| 設計判断・品質基準の定義 | 基準に従った実装・指摘 |

## 今後の示唆

> もしかしたら、我々が手を動かしてコードを書く時代ももうそろそろ終わりで、いかにうまく指示を出せるかがポイントになってくるかもしれない。

コードを書くスキルより、**AI に適切な指示を出す設計力** が重要になる時代が来ている。
